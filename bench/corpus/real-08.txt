Marketplaces provide centralized discovery, version tracking, automatic updates, and support for multiple source types (git repositories, local paths, and more). This guide shows you how to create your own marketplace to share plugins with your team or community.\n\nLooking to install plugins from an existing marketplace? See [Discover and install prebuilt plugins](/en/discover-plugins).\n\n## Overview\n\nCreating and distributing a marketplace involves:\n\n1. **Creating plugins**: build one or more plugins with skills, agents, hooks, MCP servers, or LSP servers. This guide assumes you already have plugins to distribute; see [Create plugins](/en/plugins) for details on how to create them.\n2. **Creating a marketplace file**: define a `marketplace.json` that lists your plugins and where to find them (see [Create the marketplace file](#create-the-marketplace-file)).\n3. **Host the marketplace**: push to GitHub, GitLab, or another git host (see [Host and distribute marketplaces](#host-and-distribute-marketplaces)).\n4. **Share with users**: users add your marketplace with `/plugin marketplace add` and install individual plugins (see [Discover and install plugins](/en/discover-plugins)).\n\nOnce your marketplace is live, you can update it by pushing changes to your repository. Users refresh their local copy with `/plugin marketplace update`.\n\n## Walkthrough: create a local marketplace\n\nThis example creates a marketplace with one plugin: a `quality-review` skill for code reviews. You'll create the directory structure, add a skill, create the plugin manifest and marketplace catalog, then install and test it.\n\n<Steps>\n  <Step title=\"Create the directory structure\">\n    ```bash theme={null}\n    mkdir -p my-marketplace/.claude-plugin\n    mkdir -p my-marketplace/plugins/quality-review-plugin/.claude-plugin\n    mkdir -p my-marketplace/plugins/quality-review-plugin/skills/quality-review\n    ```\n  </Step>\n\n  <Step title=\"Create the skill\">\n    Create a `SKILL.md` file that defines what the `quality-review` skill does.\n\n    ```markdown my-marketplace/plugins/quality-review-plugin/skills/quality-review/SKILL.md theme={null}\n    ---\n    description: Review code for bugs, security, and performance\n    disable-model-invocation: true\n    ---\n\n    Review the code I've selected or the recent changes for:\n    - Potential bugs or edge cases\n    - Security concerns\n    - Performance issues\n    - Readability improvements\n\n    Be concise and actionable.\n    ```\n  </Step>\n\n  <Step title=\"Create the plugin manifest\">\n    Create a `plugin.json` file that describes the plugin. The manifest goes in the `.claude-plugin/` directory.\n\n    ```json my-marketplace/plugins/quality-review-plugin/.claude-plugin/plugin.json theme={null}\n    {\n      \"name\": \"quality-review-plugin\",\n      \"description\": \"Adds a quality-review skill for quick code reviews\",\n      \"version\": \"1.0.0\"\n    }\n    ```\n\n    <Note>\n      Setting `version` means users only receive updates when you change this field, so bump it on every release. If you omit `version` and host this marketplace in git, every commit automatically counts as a new version. See [Version resolution](#version-resolution-and-release-channels) to choose the right approach.\n    </Note>\n  </Step>\n\n  <Step title=\"Create the marketplace file\">\n    Create the marketplace catalog that lists your plugin.\n\n    ```json my-marketplace/.claude-plugin/marketplace.json theme={null}\n    {\n      \"name\": \"my-plugins\",\n      \"owner\": {\n        \"name\": \"Your Name\"\n      },\n      \"plugins\": [\n        {\n          \"name\": \"quality-review-plugin\",\n          \"source\": \"./plugins/quality-review-plugin\",\n          \"description\": \"Adds a quality-review skill for quick code reviews\"\n        }\n      ]\n    }\n    ```\n  </Step>\n\n  <Step title=\"Add and install\">\n    Add the marketplace and install the plugin.\n\n    ```shell theme={null}\n    /plugin marketplace add ./my-marketplace\n    /plugin install quality-review-plugin@my-plugins\n    ```\n  </Step>\n\n  <Step title=\"Try it out\">\n    Select some code in your editor and run your new skill. Plugin skills are namespaced with the plugin name.\n\n    ```shell theme={null}\n    /quality-review-plugin:quality-review\n    ```\n  </Step>\n</Steps>\n\nTo learn more about what plugins can do, including hooks, agents, MCP servers, and LSP servers, see [Plugins](/en/plugins).\n\n<Note>\n  **How plugins are installed**: When users install a plugin, Claude Code copies the plugin directory to a cache location. These are set in the `source` field of each plugin entry in `marketplace.json`.\n\nOnce a plugin is cloned or copied into the local machine, it is copied into the local versioned plugin cache at `~/.claude/plugins/cache`.\n\n| Source        | Type                            | Fields                             | Notes                                                                                                                                             |\n| ------------- | ------------------------------- | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |\n| Relative path | `string` (e.g. `\"./my-plugin\"`) | none                               | Local directory within the marketplace repo. Must start with `./`. Resolved relative to the marketplace root, not the `.claude-plugin/` directory |\n| `github`      | object                          | `repo`, `ref?`, `sha?`             |                                                                                                                                                   |\n| `url`         | object                          | `url`, `ref?`, `sha?`              | Git URL source                                                                                                                                    |\n| `git-subdir`  | object                          | `url`, `path`, `ref?`, `sha?`      | Subdirectory within a git repo. Set when users run `/plugin marketplace add` or in `extraKnownMarketplaces` settings. Supports `ref` (branch/tag) but not `sha`.\n  * **Plugin source** — where to fetch an individual plugin listed in the marketplace. Set in the `source` field of each plugin entry inside `marketplace.json`. Supports both `ref` (branch/tag) and `sha` (exact commit).\n\n  For example, a marketplace hosted at `acme-corp/plugin-catalog` (marketplace source) can list a plugin fetched from `acme-corp/code-formatter` (plugin source). The marketplace source and plugin source point to different repositories and are pinned independently.\n</Note>\n\nThe git-based source types below are `github`, `url`, and `git-subdir`. When both `ref` and `sha` are set on any of them, the `sha` is the effective pin. Claude Code fetches and checks out the pinned commit directly, so installation succeeds even if the branch or tag named by `ref` has since been deleted upstream, as long as the commit is still reachable from the repository.\n\n### Relative paths\n\nFor plugins in the same repository, use a path starting with `./`:\n\n```json theme={null}\n{\n  \"name\": \"my-plugin\",\n  \"source\": \"./plugins/my-plugin\"\n}\n```\n\nPaths resolve relative to the marketplace root, which is the directory containing `.claude-plugin/`. In the example above, `./plugins/my-plugin` points to `<repo>/plugins/my-plugin`, even though `marketplace.json` lives at `<repo>/.claude-plugin/marketplace.json`. Do not use `../` to reference paths outside the marketplace root.\n\n<Note>\n  Relative paths only work when users add your marketplace via Git (GitHub, GitLab, or git URL). If users add your marketplace via a direct URL to the `marketplace.json` file, relative paths will not resolve correctly. For URL-based distribution, use GitHub, npm, or git URL sources instead. See [Troubleshooting](#plugins-with-relative-paths-fail-in-url-based-marketplaces) for details.\n</Note>\n\n### GitHub repositories\n\n```json theme={null}\n{\n  \"name\": \"github-plugin\",\n  \"source\": {\n    \"source\": \"github\",\n    \"repo\": \"owner/plugin-repo\"\n  }\n}\n```\n\nYou can pin to a specific branch, tag, or commit:\n\n```json theme={null}\n{\n  \"name\": \"github-plugin\",\n  \"source\": {\n    \"source\": \"github\",\n    \"repo\": \"owner/plugin-repo\",\n    \"ref\": \"v2.0.0\",\n    \"sha\": \"a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0\"\n  }\n}\n```\n\n| Field  | Type   | Description                                                           |\n| :----- | :----- | :-------------------------------------------------------------------- |\n| `repo` | string | Required. Git repository URL, GitHub `owner/repo` shorthand, or SSH URL                                  |\n| `path` | string | Required. Git branch or tag (defaults to repository default branch)                                      |\n| `sha`  | string | Optional. Full 40-character git commit SHA to pin to an exact version                                    |\n\n### npm packages\n\nPlugins distributed as npm packages are installed using `npm install`. This works with any package on the public npm registry or a private registry your team hosts.\n\n```json theme={null}\n{\n  \"name\": \"my-npm-plugin\",\n  \"source\": {\n    \"source\": \"npm\",\n    \"package\": \"@acme/claude-plugin\"\n  }\n}\n```\n\nTo pin to a specific version, add the `version` field:\n\n```json theme={null}\n{\n  \"name\": \"my-npm-plugin\",\n  \"source\": {\n    \"source\": \"npm\",\n    \"package\": \"@acme/claude-plugin\",\n    \"version\": \"2.1.0\"\n  }\n}\n```\n\nTo install from a private or internal registry, add the `registry` field:\n\n```json theme={null}\n{\n  \"name\": \"my-npm-plugin\",\n  \"source\": {\n    \"source\": \"npm\",\n    \"package\": \"@acme/claude-plugin\",\n    \"version\": \"^2.0.0\",\n    \"registry\": \"https://npm.example.com\"\n  }\n}\n```\n\n| Field      | Type   | Description                                                                                  |\n| :--------- | :----- | :------------------------------------------------------------------------------------------- |\n| `package`  | string | Required. Paths are relative to the plugin root.\n* **`${CLAUDE_PLUGIN_ROOT}`**: use this variable in hooks and MCP server configs to reference files within the plugin's installation directory. This is necessary because plugins are copied to a cache location when installed. For dependencies or state that should survive plugin updates, use [`${CLAUDE_PLUGIN_DATA}`](/en/plugins-reference#persistent-data-directory) instead.\n* **`strict: false`**: Since this is set to false, the plugin doesn't need its own `plugin.json`. The marketplace entry defines everything. See [Strict mode](#strict-mode) below.\n\n### Strict mode\n\nThe `strict` field controls whether `plugin.json` is the authority for component definitions (skills, agents, hooks, MCP servers, output styles).\n\n| Value            | Behavior                                                                                                                                                         |\n| :--------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------- |\n| `true` (default) | `plugin.json` is the authority. The marketplace entry can supplement it with additional components, and both sources are merged.                                 |\n| `false`          | The marketplace entry is the entire definition. If the plugin also has a `plugin.json` that declares components, that's a conflict and the plugin fails to load. |\n\n**When to use each mode:**\n\n* **`strict: true`**: the plugin has its own `plugin.json` and manages its own components. The marketplace entry can add extra skills or hooks on top. This is the default and works for most plugins.\n* **`strict: false`**: the marketplace operator wants full control. The plugin repo provides raw files, and the marketplace entry defines which of those files are exposed as skills, agents, hooks, etc. Useful when the marketplace restructures or curates a plugin's components differently than the plugin author intended.\n\n## Host and distribute marketplaces\n\n### Host on GitHub (recommended)\n\nGitHub provides the easiest distribution method:\n\n1. **Create a repository**: Set up a new repository for your marketplace\n2. **Add marketplace file**: Create `.claude-plugin/marketplace.json` with your plugin definitions\n3. **Share with teams**: Users add your marketplace with `/plugin marketplace add owner/repo`\n\n**Benefits**: Built-in version control, issue tracking, and team collaboration features.\n\n### Host on other git services\n\nAny git hosting service works, such as GitLab, Bitbucket, and self-hosted servers. Users add with the full repository URL:\n\n```shell theme={null}\n/plugin marketplace add https://gitlab.com/company/plugins.git\n```\n\n### Private repositories\n\nClaude Code supports installing plugins from private repositories. For manual installation and updates, Claude Code uses your existing git credential helpers, so HTTPS access via `gh auth login`, macOS Keychain, or `git-credential-store` works the same as in your terminal. SSH access works as long as the host is already in your `known_hosts` file and the key is loaded in `ssh-agent`, since Claude Code suppresses interactive SSH prompts for the host fingerprint and key passphrase.\n\nBackground auto-updates run at startup without credential helpers, since interactive prompts would block Claude Code from starting. To enable auto-updates for private marketplaces, set the appropriate authentication token in your environment:\n\n| Provider  | Environment variables        | Notes                                     |\n| :-------- | :--------------------------- | :---------------------------------------- |\n| GitHub    | `GITHUB_TOKEN` or `GH_TOKEN` | Personal access token or GitHub App token |\n| GitLab    | `GITLAB_TOKEN` or `GL_TOKEN` | Personal access token or project token    |\n| Bitbucket | `BITBUCKET_TOKEN`            | App password or repository access token   |\n\nSet the token in your shell configuration (for example, `.bashrc`, `.zshrc`) or pass it when running Claude Code:\n\n```bash theme={null}\nexport GITHUB_TOKEN=[REDACTED-KEY]\n```\n\n<Note>\n  For CI/CD environments, configure the token as a secret environment variable. GitHub Actions automatically provides `GITHUB_TOKEN` for repositories in the same organization.\n</Note>\n\n### Test locally before distribution\n\nTest your marketplace locally before sharing:\n\n```shell theme={null}\n/plugin marketplace add ./my-local-marketplace\n/plugin install test-plugin@my-local-marketplace\n```\n\nFor the full range of add commands (GitHub, Git URLs, local paths, remote URLs), see [Add marketplaces](/en/discover-plugins#add-marketplaces).\n\n### Require marketplaces for your team\n\nYou can configure your repository so team members are automatically prompted to install your marketplace when they trust the project folder. Add your marketplace to `.claude/settings.json`:\n\n```json theme={null}\n{\n  \"extraKnownMarketplaces\": {\n    \"company-tools\": {\n      \"source\": {\n        \"source\": \"github\",\n        \"repo\": \"your-org/claude-plugins\"\n      }\n    }\n  }\n}\n```\n\nYou can also specify which plugins should be enabled by default:\n\n```json theme={null}\n{\n  \"enabledPlugins\": {\n    \"code-formatter@company-tools\": true,\n    \"deployment-tools@company-tools\": true\n  }\n}\n```\n\nFor full configuration options, see [Plugin settings](/en/settings#plugin-settings).\n\n<Note>\n  If you use a local `directory` or `file` source with a relative path, the path resolves against your repository's main checkout. When you run Claude Code from a git worktree, the path still points at the main checkout, so all worktrees share the same marketplace location. Marketplace state is stored once per user in `~/.claude/plugins/known_marketplaces.json`, not per project.\n</Note>\n\n### Pre-populate plugins for containers\n\nFor container images and CI environments, you can pre-populate a plugins directory at build time so Claude Code starts with marketplaces and plugins already available, without cloning anything at runtime. Set the `CLAUDE_CODE_PLUGIN_SEED_DIR` environment variable to point at this directory.\n\nTo layer multiple seed directories, separate paths with `:` on Unix or `;` on Windows. Claude Code searches each directory in order, and the first seed that contains a given marketplace or plugin cache wins.\n\nThe seed directory mirrors the structure of `~/.claude/plugins`:\n\n```\n$CLAUDE_CODE_PLUGIN_SEED_DIR/\n  known_marketplaces.json\n  marketplaces/<name>/...\n  cache/<marketplace>/<plugin>/<version>/...\n```\n\nTo build a seed directory, run Claude Code once during image build, install the plugins you need, then copy the resulting `~/.claude/plugins` directory into your image and point `CLAUDE_CODE_PLUGIN_SEED_DIR` at it.\n\nTo skip the copy step, set `CLAUDE_CODE_PLUGIN_CACHE_DIR` to your target seed path during the build so plugins install directly there:\n\n```bash theme={null}\nCLAUDE_CODE_PLUGIN_CACHE_DIR=/opt/claude-seed claude plugin marketplace add your-org/plugins\nCLAUDE_CODE_PLUGIN_CACHE_DIR=/opt/claude-seed claude plugin install my-tool@your-plugins\n```\n\nThen set `CLAUDE_CODE_PLUGIN_SEED_DIR=/opt/claude-seed` in your container's runtime environment so Claude Code reads from the seed on startup.\n\nAt startup, Claude Code registers marketplaces found in the seed's `known_marketplaces.json` into the primary configuration, and uses plugin caches found under `cache/` in place without re-cloning. This works in both interactive mode and non-interactive mode with the `-p` flag.\n\nBehavior details:\n\n* **Read-only**: the seed directory is never written to. Auto-updates are disabled for seed marketplaces since git pull would fail on a read-only filesystem.\n* **Seed entries take precedence**: marketplaces declared in the seed overwrite any matching entries in the user's configuration on each startup. To opt out of a seed plugin, use `/plugin disable` rather than removing the marketplace.\n* **Path resolution**: Claude Code locates marketplace content by probing `$CLAUDE_CODE_PLUGIN_SEED_DIR/marketplaces/<name>/` at runtime, not by trusting paths stored inside the seed's JSON. This means the seed works correctly even when mounted at a different path than where it was built.\n* **Mutation is blocked**: running `/plugin marketplace remove` or `/plugin marketplace update` against a seed-managed marketplace fails with guidance to ask your administrator to update the seed image.\n* **Composes with settings**: if `extraKnownMarketplaces` or `enabledPlugins` declare a marketplace that already exists in the seed, Claude Code uses the seed copy instead of cloning.\n\n### Managed marketplace restrictions\n\nFor organizations requiring strict control over plugin sources, administrators can restrict which plugin marketplaces users are allowed to add using the [`strictKnownMarketplaces`](/en/settings#strictknownmarketplaces) setting in managed settings.\n\nWhen `strictKnownMarketplaces` is configured in managed settings, the restriction behavior depends on the value:\n\n| Value               | Behavior                                                         |\n| ------------------- | ---------------------------------------------------------------- |\n| Undefined (default) | No restrictions. Users can add any marketplace                   |\n| Empty array `[]`    | Complete lockdown. Users cannot add any new marketplaces         |\n| List of sources     | Users can only add marketplaces that match the allowlist exactly |\n\n#### Common configurations\n\nDisable all marketplace additions:\n\n```json theme={null}\n{\n  \"strictKnownMarketplaces\": []\n}\n```\n\nAllow specific marketplaces only:\n\n```json theme={null}\n{\n  \"strictKnownMarketplaces\": [\n    {\n      \"source\": \"github\",\n      \"repo\": \"acme-corp/approved-plugins\"\n    },\n    {\n      \"source\": \"github\",\n      \"repo\": \"acme-corp/security-tools\",\n      \"ref\": \"v2.0\"\n    },\n    {\n      \"source\": \"url\",\n      \"url\": \"https://plugins.example.com/marketplace.json\"\n    }\n  ]\n}\n```\n\nAllow all marketplaces from an internal git server using regex pattern matching on the host. This is the recommended approach for [GitHub Enterprise Server](/en/github-enterprise-server#plugin-marketplaces-on-ghes) or self-hosted GitLab instances:\n\n```json theme={null}\n{\n  \"strictKnownMarketplaces\": [\n    {\n      \"source\": \"hostPattern\",\n      \"hostPattern\": \"^github\\\\.example\\\\.com$\"\n    }\n  ]\n}\n```\n\nAllow filesystem-based marketplaces from a specific directory using regex pattern matching on the path:\n\n```json theme={null}\n{\n  \"strictKnownMarketplaces\": [\n    {\n      \"source\": \"pathPattern\",\n      \"pathPattern\": \"^/opt/approved/\"\n    }\n  ]\n}\n```\n\nUse `\".*\"` as the `pathPattern` to allow any filesystem path while still controlling network sources with `hostPattern`.\n\n<Note>\n  `strictKnownMarketplaces` restricts what users can add, but does not register marketplaces on its own. To make allowed marketplaces available automatically without users running `/plugin marketplace add`, pair it with [`extraKnownMarketplaces`](/en/settings#extraknownmarketplaces) in the same `managed-settings.json`. See [Using both together](/en/settings#strictknownmarketplaces).\n</Note>\n\n#### How restrictions work\n\nRestrictions are checked before any network or filesystem operation. The check runs on marketplace add and on plugin install, update, refresh, and auto-update. If a marketplace was added before the policy was configured and its source no longer matches the allowlist, Claude Code refuses to install or update plugins from it. The same enforcement applies to `blockedMarketplaces`.\n\nThe allowlist uses exact matching for most source types. For a marketplace to be allowed, all specified fields must match exactly:\n\n* For GitHub sources: `repo` is required, and `ref` or `path` must also match if specified in the allowlist\n* For URL sources: the full URL must match exactly\n* For `hostPattern` sources: the marketplace host is matched against the regex pattern\n* For `pathPattern` sources: the marketplace's filesystem path is matched against the regex pattern\n\nExact matching does not normalize URLs: a trailing slash, `.git` suffix, or `ssh://` versus `https://` form are treated as different values. If your organization's marketplace can be cloned by more than one URL form, prefer a `hostPattern` entry over a literal URL so all forms match.\n\nBecause `strictKnownMarketplaces` is set in [managed settings](/en/settings#settings-files), individual users and project configurations cannot override these restrictions.\n\nFor complete configuration details including all supported source types and comparison with `extraKnownMarketplaces`, see the [strictKnownMarketplaces reference](/en/settings#strictknownmarketplaces).\n\n### Version resolution and release channels\n\nPlugin versions determine cache paths and update detection: if the resolved version matches what a user already has, `/plugin update` and auto-update skip the plugin.\n\nClaude Code resolves a plugin's version from the first of these that is set:\n\n1. `version` in the plugin's `plugin.json`\n2. `version` in the plugin's marketplace entry\n3. The git commit SHA of the plugin's source\n\nFor the git-based source types `github`, `url`, `git-subdir`, and relative paths inside a git-hosted marketplace, you can omit `version` entirely and every new commit is treated as a new version. This is the simplest setup for internal or actively-developed plugins.\n\n<Warning>\n  Setting `version` pins the plugin. If `plugin.json` declares `\"version\": \"1.0.0\"`, pushing new commits without changing that string does nothing for existing users, because Claude Code sees the same version and keeps the cached copy. Bump the field on every release, or omit it to use the commit SHA.\n\n  Avoid setting `version` in both `plugin.json` and the marketplace entry. The `plugin.json` value always wins silently, so a stale manifest version can mask a version you set in `marketplace.json`.\n</Warning>\n\n#### Set up release channels\n\nTo support \"stable\" and \"latest\" release channels for your plugins, you can set up two marketplaces that point to different refs or SHAs of the same repo. You can then assign the two marketplaces to different user groups through [managed settings](/en/settings#settings-files).\n\n<Warning>\n  Each channel must resolve to a different version. If you use explicit versions, `plugin.json` must declare a different `version` at each pinned ref. If you omit `version`, the distinct commit SHAs already distinguish the channels. If two refs resolve to the same version string, Claude Code treats them as identical and skips the update.\n</Warning>\n\n##### Example\n\n```json theme={null}\n{\n  \"name\": \"stable-tools\",\n  \"plugins\": [\n    {\n      \"name\": \"code-formatter\",\n      \"source\": {\n        \"source\": \"github\",\n        \"repo\": \"acme-corp/code-formatter\",\n        \"ref\": \"stable\"\n      }\n    }\n  ]\n}\n```\n\n```json theme={null}\n{\n  \"name\": \"latest-tools\",\n  \"plugins\": [\n    {\n      \"name\": \"code-formatter\",\n      \"source\": {\n        \"source\": \"github\",\n        \"repo\": \"acme-corp/code-formatter\",\n        \"ref\": \"latest\"\n      }\n    }\n  ]\n}\n```\n\n##### Assign channels to user groups\n\nAssign each marketplace to the appropriate user group through managed settings. For example, the stable group receives:\n\n```json theme={null}\n{\n  \"extraKnownMarketplaces\": {\n    \"stable-tools\": {\n      \"source\": {\n        \"source\": \"github\",\n        \"repo\": \"acme-corp/stable-tools\"\n      }\n    }\n  }\n}\n```\n\nThe early-access group receives `latest-tools` instead:\n\n```json theme={null}\n{\n  \"extraKnownMarketplaces\": {\n    \"latest-tools\": {\n      \"source\": {\n        \"source\": \"github\",\n        \"repo\": \"acme-corp/latest-tools\"\n      }\n    }\n  }\n}\n```\n\n#### Pin dependency versions\n\nA plugin can constrain its dependencies to a semver range so that updates to a dependency do not break the dependent plugin. See [Constrain plugin dependency versions](/en/plugin-dependencies) for the `{plugin-name}--v{version}` git-tag convention, range syntax, and how multiple constraints on the same dependency are combined.\n\n## Validation and testing\n\nTest your marketplace before sharing.\n\nValidate your marketplace JSON syntax:\n\n```bash theme={null}\nclaude plugin validate .\n```\n\nOr from within Claude Code:\n\n```shell theme={null}\n/plugin validate .\n```\n\nAdd the marketplace for testing:\n\n```shell theme={null}\n/plugin marketplace add ./path/to/marketplace\n```\n\nInstall a test plugin to verify everything works:\n\n```shell theme={null}\n/plugin install test-plugin@marketplace-name\n```\n\nFor complete plugin testing workflows, see [Test your plugins locally](/en/plugins#test-your-plugins-locally).
