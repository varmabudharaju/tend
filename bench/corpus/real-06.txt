The only valid `isolation` value is `\"worktree\"`. For security reasons, `hooks`, `mcpServers`, and `permissionMode` are not supported for plugin-shipped agents.\n\n**Integration points**:\n\n* Agents appear in the `/agents` interface\n* Claude can invoke agents automatically based on task context\n* Agents can be invoked manually by users\n* Plugin agents work alongside built-in Claude agents\n\nFor complete details, see [Subagents](/en/sub-agents).\n\n### Hooks\n\nPlugins can provide event handlers that respond to Claude Code events automatically.\n\n**Location**: `hooks/hooks.json` in plugin root, or inline in plugin.json\n\n**Format**: JSON configuration with event matchers and actions\n\n**Hook configuration**:\n\n```json theme={null}\n{\n  \"hooks\": {\n    \"PostToolUse\": [\n      {\n        \"matcher\": \"Write|Edit\",\n        \"hooks\": [\n          {\n            \"type\": \"command\",\n            \"command\": \"\\\"${CLAUDE_PLUGIN_ROOT}\\\"/scripts/format-code.sh\"\n          }\n        ]\n      }\n    ]\n  }\n}\n```\n\nPlugin hooks respond to the same lifecycle events as [user-defined hooks](/en/hooks):\n\n| Event                 | When it fires                                                                                                                                          |\n| :-------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------- |\n| `SessionStart`        | When a session begins or resumes                                                                                                                       |\n| `Setup`               | When you start Claude Code with `--init-only`, or with `--init` or `--maintenance` in `-p` mode. Each monitor runs a shell command for the lifetime of the session and delivers every stdout line to Claude as a notification, so Claude can react to log entries, status changes, or polled events without being asked to start the watch itself.\n\nPlugin monitors use the same mechanism as the [Monitor tool](/en/tools-reference#monitor-tool) and share its availability constraints. They run only in interactive CLI sessions, run unsandboxed at the same trust level as [hooks](#hooks), and are skipped on hosts where the Monitor tool is unavailable.\n\n<Note>\n  Plugin monitors require Claude Code v2.1.105 or later.\n</Note>\n\n**Location**: `monitors/monitors.json` in the plugin root, or inline in `plugin.json`\n\n**Format**: JSON array of monitor entries\n\nThe following `monitors/monitors.json` watches a deployment status endpoint and a local error log:\n\n```json theme={null}\n[\n  {\n    \"name\": \"deploy-status\",\n    \"command\": \"\\\"${CLAUDE_PLUGIN_ROOT}\\\"/scripts/poll-deploy.sh ${user_config.api_endpoint}\",\n    \"description\": \"Deployment status changes\"\n  },\n  {\n    \"name\": \"error-log\",\n    \"command\": \"tail -F ./logs/error.log\",\n    \"description\": \"Application error log\",\n    \"when\": \"on-skill-invoke:debug\"\n  }\n]\n```\n\nTo declare monitors inline, set `experimental.monitors` in `plugin.json` to the same array. To load from a non-default path, set `experimental.monitors` to a relative path string such as `\"./config/monitors.json\"`. Monitors are an [experimental component](#experimental-components).\n\n**Required fields:**\n\n| Field         | Description                                                                                                           |\n| :------------ | :-------------------------------------------------------------------------------------------------------------------- |\n| `name`        | Identifier unique within the plugin. Prevents duplicate processes when the plugin reloads or a skill is invoked again |\n| `command`     | Shell command run as a persistent background process in the session working directory                                 |\n| `description` | Short summary of what is being watched. Shown in the task panel and in notification summaries                         |\n\n**Optional fields:**\n\n| Field  | Description                                                                                                                                                                                                              |\n| :----- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |\n| `when` | Controls when the monitor starts. `\"always\"` starts it at session start and on plugin reload, and is the default. `\"on-skill-invoke:<skill-name>\"` starts it the first time the named skill in this plugin is dispatched |\n\nThe `command` value supports the same [variable substitutions](#environment-variables) as MCP and LSP server configs: `${CLAUDE_PLUGIN_ROOT}`, `${CLAUDE_PLUGIN_DATA}`, `${CLAUDE_PROJECT_DIR}`, `${user_config.*}`, and any `${ENV_VAR}` from the environment. Prefix the command with `cd \"${CLAUDE_PLUGIN_ROOT}\" && ` if the script needs to run from the plugin's own directory.\n\nDisabling a plugin mid-session does not stop monitors that are already running. They stop when the session ends.\n\n### Themes\n\nPlugins can ship color themes that appear in `/theme` alongside the built-in presets and the user's local themes. A theme is a JSON file in `themes/` with a `base` preset and a sparse `overrides` map of color tokens. Themes are an [experimental component](#experimental-components).\n\n```json theme={null}\n{\n  \"name\": \"Dracula\",\n  \"base\": \"dark\",\n  \"overrides\": {\n    \"claude\": \"#bd93f9\",\n    \"error\": \"#ff5555\",\n    \"success\": \"#50fa7b\"\n  }\n}\n```\n\nSelecting a plugin theme persists `custom:<plugin-name>:<slug>` in the user's config. Plugin themes are read-only; pressing `Ctrl+E` on one in `/theme` copies it into `~/.claude/themes/` so the user can edit the copy.\n\n***\n\n## Plugin installation scopes\n\nWhen you install a plugin, you choose a **scope** that determines where the plugin is available and who else can use it:\n\n| Scope     | Settings file                                   | Use case                                                 |\n| :-------- | :---------------------------------------------- | :------------------------------------------------------- |\n| `user`    | `~/.claude/settings.json`                       | Personal plugins available across all projects (default) |\n| `project` | `.claude/settings.json`                         | Team plugins shared via version control                  |\n| `local`   | `.claude/settings.local.json`                   | Project-specific plugins, gitignored                     |\n| `managed` | [Managed settings](/en/settings#settings-files) | Managed plugins (read-only, update only)                 |\n\nPlugins use the same scope system as other Claude Code configurations. For installation instructions and scope flags, see [Install plugins](/en/discover-plugins#install-plugins). For a complete explanation of scopes, see [Configuration scopes](/en/settings#configuration-scopes).\n\n***\n\n## Skills-directory plugins\n\nAny folder under a skills directory that contains a `.claude-plugin/plugin.json` manifest is loaded as a plugin named `<name>@skills-dir` on the next session, with no marketplace and no install step. Scaffold one with [`plugin init`](#plugin-init). Unlike a marketplace install, the plugin is discovered in place rather than copied into the plugin cache.\n\nA skills directory tree supports three distinct things:\n\n| What you have                                 | What it is                                                                          |\n| :-------------------------------------------- | :---------------------------------------------------------------------------------- |\n| `<skills-dir>/foo/SKILL.md` with no manifest  | A plain [skill](/en/skills) named `foo`                                             |\n| `<skills-dir>/foo/.claude-plugin/plugin.json` | A plugin `foo@skills-dir`, which can bundle its own skills, agents, hooks, and more |\n| `<plugin>/skills/bar/SKILL.md`                | A skill `bar` packaged inside a plugin                                              |\n\n### Choose where the plugin loads from\n\n| Skills directory        | Scope    | Loads                                                                            |\n| :---------------------- | :------- | :------------------------------------------------------------------------------- |\n| `~/.claude/skills/`     | personal | In every project, since the location is yours alone                              |\n| `<cwd>/.claude/skills/` | project  | Only after you accept the workspace [trust dialog](/en/settings) for that folder |\n\nA project-scope plugin is checked into the repository and reaches every collaborator who clones it. Because that content comes from the repository rather than from you, it loads only after the same trust gate that governs `.claude/settings.json`, and components that run code are restricted further:\n\n* MCP servers it declares go through the [same per-server approval](/en/mcp) as a project `.mcp.json`\n* LSP servers start only after you trust the workspace\n* [Background monitors](#monitors) do not load\n\nPersonal-scope plugins have none of these restrictions.\n\n<Warning>\n  Project-scope `@skills-dir` plugins load only from the `.claude/skills/` of the directory where you start Claude Code. They do not [walk up to the repository root](/en/skills#automatic-discovery-from-parent-and-nested-directories) the way plain skills and commands do, so launching from a subdirectory misses a plugin that lives at the repo root. Launch from the repository root, or run `/reload-plugins` after changing directories.\n</Warning>\n\n### Edit, reload, and disable a skills-directory plugin\n\nChanges you make to a skill's `SKILL.md` take effect immediately in the current session. Changes to the plugin's other components, such as `hooks/`, `.mcp.json`, `agents/`, and `output-styles/`, do not. Run `/reload-plugins` or restart Claude Code to pick those up. See [Live change detection](/en/skills#live-change-detection).\n\nTo stop loading a skills-directory plugin, delete its folder or disable it by name. There is no `uninstall` step because nothing was installed from a marketplace.\n\n```bash theme={null}\nclaude plugin disable my-tool@skills-dir\n```\n\n***\n\n## Plugin manifest schema\n\nThe `.claude-plugin/plugin.json` file defines your plugin's metadata and configuration. This section documents all supported fields and options.\n\nThe manifest is optional. If omitted, Claude Code auto-discovers components in [default locations](#file-locations-reference) and derives the plugin name from the directory name. Use a manifest when you need to provide metadata or custom component paths.\n\n### Complete schema\n\n```json theme={null}\n{\n  \"name\": \"plugin-name\",\n  \"displayName\": \"Plugin Name\",\n  \"version\": \"1.2.0\",\n  \"description\": \"Brief plugin description\",\n  \"author\": {\n    \"name\": \"Author Name\",\n    \"email\": \"redacted@example.com\",\n    \"url\": \"https://github.com/author\"\n  },\n  \"homepage\": \"https://docs.example.com/plugin\",\n  \"repository\": \"https://github.com/author/plugin\",\n  \"license\": \"MIT\",\n  \"keywords\": [\"keyword1\", \"keyword2\"],\n  \"skills\": \"./custom/skills/\",\n  \"commands\": [\"./custom/commands/special.md\"],\n  \"agents\": [\"./custom/agents/reviewer.md\"],\n  \"hooks\": \"./config/hooks.json\",\n  \"mcpServers\": \"./mcp-config.json\",\n  \"outputStyles\": \"./styles/\",\n  \"lspServers\": \"./.lsp.json\",\n  \"experimental\": {\n    \"themes\": \"./themes/\",\n    \"monitors\": \"./monitors.json\"\n  },\n  \"dependencies\": [\n    \"helper-lib\",\n    { \"name\": \"secrets-vault\", \"version\": \"~2.1.0\" }\n  ]\n}\n```\n\n### Required fields\n\nIf you include a manifest, `name` is the only required field.\n\n| Field  | Type   | Description                               | Example              |\n| :----- | :----- | :---------------------------------------- | :------------------- |\n| `name` | string | Unique identifier (kebab-case, no spaces) | `\"deployment-tools\"` |\n\nThis name is used for namespacing components. You can keep\nmetadata from another ecosystem in `plugin.json` and the plugin still loads.\nThis makes it practical to maintain one manifest that doubles as a VS Code or\nCursor extension manifest, an npm `package.json`, or an MCPB/DXT bundle\nmanifest.\n\n`claude plugin validate` reports unrecognized fields as warnings, not errors.\nIf a field is one or two characters off from a recognized one, the warning\nsuggests the likely intended name. The user turns it on with `claude plugin enable <plugin>` or the `/plugin` interface. Use this for plugins that add cost or scope a user should opt into, such as one that connects to an external service. This requires Claude Code v2.1.154 or later. Earlier versions ignore the field and enable the plugin on install.\n\n`defaultEnabled` is the fallback when nothing else has decided the plugin's state. Two things take precedence over it:\n\n* **The user's setting**: an entry for the plugin in `enabledPlugins` at any settings scope. Once written, it persists across plugin updates and reinstalls, so changing `defaultEnabled` in a later release does not flip an existing user.\n* **A dependency requirement**: when a plugin is required by another one that is active, Claude Code writes `true` for it at install or enable time. That gives it an explicit setting, so its own default no longer applies. See [Enable or disable a plugin with dependencies](/en/plugin-dependencies#enable-or-disable-a-plugin-with-dependencies).\n\nThe same field can appear in a plugin's marketplace entry, where it takes precedence over the value in `plugin.json`. See [User configuration](#user-configuration)                                                           | See below                                            |\n| `channels`              | array                 | Channel declarations for message injection (Telegram, Slack, Discord style). See [Channels](#channels)                                                    | See below                                            |\n| `dependencies`          | array                 | Other plugins this plugin requires, optionally with semver version constraints. See [Constrain plugin dependency versions](/en/plugin-dependencies)       | `[{ \"name\": \"secrets-vault\", \"version\": \"~2.1.0\" }]` |\n\n### Experimental components\n\nComponents under the `experimental` key, `themes` and `monitors`, have a manifest schema that may change between releases while they stabilize. Where you declare them is a separate migration: the top level still works, `claude plugin validate` warns, and a future release will require `experimental.*`.\n\n### User configuration\n\nThe `userConfig` field declares values that Claude Code prompts the user for when the plugin is enabled. Use this instead of requiring users to hand-edit `settings.json`.\n\n```json theme={null}\n{\n  \"userConfig\": {\n    \"api_endpoint\": {\n      \"type\": \"string\",\n      \"title\": \"API endpoint\",\n      \"description\": \"Your team's API endpoint\"\n    },\n    \"api_token\": {\n      \"type\": \"string\",\n      \"title\": \"API token\",\n      \"description\": \"API authentication token\",\n      \"sensitive\": true\n    }\n  }\n}\n```\n\nKeys must be valid identifiers. Each option supports these fields:\n\n| Field         | Required | Description                                                                              |\n| :------------ | :------- | :--------------------------------------------------------------------------------------- |\n| `type`        | Yes      | One of `string`, `number`, `boolean`, `directory`, or `file`                             |\n| `title`       | Yes      | Label shown in the configuration dialog                                                  |\n| `description` | Yes      | Help text shown beneath the field                                                        |\n| `sensitive`   | No       | If `true`, masks input and stores the value in secure storage instead of `settings.json` |\n| `required`    | No       | If `true`, validation fails when the field is empty                                      |\n| `default`     | No       | Value used when the user provides nothing                                                |\n| `multiple`    | No       | For `string` type, allow an array of strings                                             |\n| `min` / `max` | No       | Bounds for `number` type                                                                 |\n\nEach value is available for substitution as `${user_config.KEY}` in MCP and LSP server configs, hook commands, and monitor commands. Non-sensitive values can also be substituted in skill and agent content. All values are exported to plugin subprocesses as `CLAUDE_PLUGIN_OPTION_<KEY>` environment variables.\n\nNon-sensitive values are stored in `settings.json` under `pluginConfigs[<plugin-id>].options`. Sensitive values go to the system keychain (or `~/.claude/.credentials.json` where the keychain is unavailable). Keychain storage is shared with OAuth tokens and has an approximately 2 KB total limit, so keep sensitive values small.\n\n### Channels\n\nThe `channels` field lets a plugin declare one or more message channels that inject content into the conversation. Each channel binds to an MCP server that the plugin provides.\n\n```json theme={null}\n{\n  \"channels\": [\n    {\n      \"server\": \"telegram\",\n      \"userConfig\": {\n        \"bot_token\": {\n          \"type\": \"string\",\n          \"title\": \"Bot token\",\n          \"description\": \"Telegram bot token\",\n          \"sensitive\": true\n        },\n        \"owner_id\": {\n          \"type\": \"string\",\n          \"title\": \"Owner ID\",\n          \"description\": \"Your Telegram user ID\"\n        }\n      }\n    }\n  ]\n}\n```\n\nThe `server` field is required and must match a key in the plugin's `mcpServers`. The optional per-channel `userConfig` uses the same schema as the top-level field, letting the plugin prompt for bot tokens or owner IDs when the plugin is enabled.\n\n### Path behavior rules\n\nWhether a custom path replaces or extends the plugin's default directory depends on the field:\n\n* **Replaces the default**: `commands`, `agents`, `outputStyles`, `experimental.themes`, `experimental.monitors`. For example, when the manifest specifies `commands`, the default `commands/` directory is not scanned. To keep the default and add more, list it explicitly: `\"commands\": [\"./commands/\", \"./extras/\"]`\n* **Adds to the default**: `skills`. The default `skills/` directory is always scanned, and directories listed in `skills` are loaded alongside it\n* **Own merge rules**: [hooks](#hooks), [MCP servers](#mcp-servers), and [LSP servers](#lsp-servers). See each section for how multiple sources combine\n\nWhen a plugin has both a default folder and the matching manifest key, Claude Code v2.1.140 and later flags the ignored folder in `/doctor`, `claude plugin list`, and the `/plugin` detail view. The plugin still loads using the manifest paths. No warning is shown when the manifest key points into the default folder, for example `\"commands\": [\"./commands/deploy.md\"]`, because the folder is addressed explicitly in that case.\n\nFor all path fields:\n\n* All paths must be relative to the plugin root and start with `./`\n* Components from custom paths use the same naming and namespacing rules\n* Multiple paths can be specified as arrays\n* When a skill path points to a directory that contains a `SKILL.md` directly, for example `\"skills\": [\"./\"]` pointing to the plugin root, the frontmatter `name` field in `SKILL.md` determines the skill's invocation name. This gives a stable name regardless of the install directory. If `name` is not set in the frontmatter, the directory basename is used as a fallback.\n\nA plugin that has a `SKILL.md` at its root, no `skills/` subdirectory, and no `skills` manifest field is automatically loaded as a single-skill plugin in Claude Code v2.1.142 and later. You do not need to set `\"skills\": [\"./\"]` in `plugin.json` for this layout. The skill's invocation name follows the same rule as above: the frontmatter `name` field, or the directory basename as a fallback.\n\n**Path examples**:\n\n```json theme={null}\n{\n  \"commands\": [\n    \"./specialized/deploy.md\",\n    \"./utilities/batch-process.md\"\n  ],\n  \"agents\": [\n    \"./custom-agents/reviewer.md\",\n    \"./custom-agents/tester.md\"\n  ]\n}\n```\n\n### Environment variables\n\nClaude Code provides three variables for referencing paths. All are substituted inline anywhere they appear in skill content, agent content, hook commands, monitor commands, and MCP or LSP server configs. All are also exported as environment variables to hook processes and MCP or LSP server subprocesses.\n\n**`${CLAUDE_PLUGIN_ROOT}`**: the absolute path to your plugin's installation directory. Use this to reference scripts, binaries, and config files bundled with the plugin. In hook commands, use [exec form](/en/hooks#exec-form-and-shell-form) with `args` so the path is passed as one argument with no quoting. In shell-form hooks and monitor commands, wrap it in double quotes, as in `\"${CLAUDE_PLUGIN_ROOT}\"`. This path changes when the plugin updates. The previous version's directory remains on disk for about seven days after an update before cleanup, but treat it as ephemeral and do not write state here.\n\nWhen a plugin updates mid-session, hook commands, monitors, MCP servers, and LSP servers keep using the previous version's path. Run `/reload-plugins` to switch hooks, MCP servers, and LSP servers to the new path; monitors require a session restart.\n\n**`${CLAUDE_PLUGIN_DATA}`**: a persistent directory for plugin state that survives updates. Use this for installed dependencies such as `node_modules` or Python virtual environments, generated code, caches, and any other files that should persist across plugin versions. The directory is created automatically the first time this variable is referenced.\n\n**`${CLAUDE_PROJECT_DIR}`**: the project root. This is the same directory hooks receive in their `CLAUDE_PROJECT_DIR` variable. Use this to reference project-local scripts or config files. Wrap in quotes to handle paths with spaces, for example `\"${CLAUDE_PROJECT_DIR}/scripts/server.sh\"`. MCP servers can also call the MCP `roots/list` request, which returns the directory Claude Code was launched from.\n\n```json theme={null}\n{\n  \"hooks\": {\n    \"PostToolUse\": [\n      {\n        \"hooks\": [\n          {\n            \"type\": \"command\",\n            \"command\": \"\\\"${CLAUDE_PLUGIN_ROOT}\\\"/scripts/process.sh\"\n          }\n        ]\n      }\n    ]\n  }\n}\n```\n\n#### Persistent data directory\n\nThe `${CLAUDE_PLUGIN_DATA}` directory resolves to `~/.claude/plugins/data/{id}/`, where `{id}` is the plugin identifier with characters outside `a-z`, `A-Z`, `0-9`, `_`, and `-` replaced by `-`. For a plugin installed as `formatter@my-marketplace`, the directory is `~/.claude/plugins/data/formatter-my-marketplace/`.\n\nA common use is installing language dependencies once and reusing them across sessions and plugin updates. Because the data directory outlives any single plugin version, a check for directory existence alone cannot detect when an update changes the plugin's dependency manifest. The recommended pattern compares the bundled manifest against a copy in the data directory and reinstalls when they differ.\n\nThis `SessionStart` hook installs `node_modules` on the first run and again whenever a plugin update includes a changed `package.json`:\n\n```json theme={null}\n{\n  \"hooks\": {\n    \"SessionStart\": [\n      {\n        \"hooks\": [\n          {\n            \"type\": \"command\",\n            \"command\": \"diff -q \\\"${CLAUDE_PLUGIN_ROOT}/package.json\\\" \\\"${CLAUDE_PLUGIN_DATA}/package.json\\\" >/dev/null 2>&1 || (cd \\\"${CLAUDE_PLUGIN_DATA}\\\" && cp \\\"${CLAUDE_PLUGIN_ROOT}/package.json\\\" . && npm install) || rm -f \\\"${CLAUDE_PLUGIN_DATA}/package.json\\\"\"\n          }\n        ]\n      }\n    ]\n  }\n}\n```\n\nThe `diff` exits nonzero when the stored copy is missing or differs from the bundled one, covering both first run and dependency-changing updates. If `npm install` fails, the trailing `rm` removes the copied manifest so the next session retries.\n\nScripts bundled in `${CLAUDE_PLUGIN_ROOT}` can then run against the persisted `node_modules`:\n\n```json theme={null}\n{\n  \"mcpServers\": {\n    \"routines\": {\n      \"command\": \"node\",\n      \"args\": [\"${CLAUDE_PLUGIN_ROOT}/server.js\"],\n      \"env\": {\n        \"NODE_PATH\": \"${CLAUDE_PLUGIN_DATA}/node_modules\"\n      }\n    }\n  }\n}\n```\n\nThe data directory is deleted automatically when you uninstall the plugin from the last scope where it is installed. The `/plugin` interface shows the directory size and prompts before deleting. The CLI deletes by default; pass [`--keep-data`](#plugin-uninstall) to preserve it.\n\n***\n\n## Plugin caching and file resolution\n\nPlugins are specified in one of two ways:\n\n* Through `claude --plugin-dir` or `claude --plugin-url`, for the duration of a session.\n* Through a marketplace, installed for future sessions.\n\nFor security and verification purposes, Claude Code copies *marketplace* plugins to the user's local **plugin cache** (`~/.claude/plugins/cache`) rather than using them in-place. Understanding this behavior is important when developing plugins that reference external files.\n\nEach installed version is a separate directory in the cache. When you update or uninstall a plugin, the previous version directory is marked as orphaned and removed automatically 7 days later. The grace period lets concurrent Claude Code sessions that already loaded the old version keep running without errors.\n\nClaude's Glob and Grep tools skip orphaned version directories during searches, so file results don't include outdated plugin code.\n\n### Path traversal limitations\n\nInstalled plugins cannot reference files outside their directory. Paths that traverse outside the plugin root (such as `../shared-utils`) will not work after installation because those external files are not copied to the cache.\n\n### Share files within a marketplace with symlinks\n\nIf your plugin needs to share files with other parts of the same marketplace, you can create symbolic links inside your plugin directory. How a symlink is handled when the plugin is copied into the cache depends on where its target resolves:\n\n* **Within the plugin's own directory:** the symlink is preserved as a relative symlink in the cache, so it keeps resolving to the copied target at runtime.\n* **Elsewhere within the same marketplace:** the symlink is dereferenced. The target's content is copied into the cache in its place. This lets a meta-plugin's `skills/` directory link to skills defined by other plugins in the marketplace.\n* **Outside the marketplace:** the symlink is skipped for security. This prevents plugins from pulling arbitrary host files such as system paths into the cache.\n\nFor plugins installed with `--plugin-dir` or from a local path, only symlinks that resolve within the plugin's own directory are preserved. All others are skipped.\n\nThe following command creates a link from inside a marketplace plugin to a shared skill defined by a sibling plugin. On Windows, use `mklink /D` from an elevated Command Prompt or enable Developer Mode:\n\n```bash theme={null}\nln -s ../../shared-plugin/skills/foo ./skills/foo\n```\n\nThis provides flexibility while maintaining the security benefits of the caching system.\n\n***\n\n## Plugin directory structure\n\n### Standard plugin layout\n\nA complete plugin follows this structure:\n\n```text theme={null}\nenterprise-plugin/\n├── .claude-plugin/           # Metadata directory (optional)\n│   └── plugin.json             # plugin manifest\n├── skills/                   # Skills\n│   ├── code-reviewer/\n│   │   └── SKILL.md\n│   └── pdf-processor/\n│       ├── SKILL.md\n│       └── scripts/\n├── commands/                 # Skills as flat .md files\n│   ├── status.md\n│   └── logs.md\n├── agents/                   # Subagent definitions\n│   ├── security-reviewer.md\n│   ├── performance-tester.md\n│   └── compliance-checker.md\n├── output-styles/            # Output style definitions\n│   └── terse.md\n├── themes/                   # Color theme definitions\n│   └── dracula.json\n├── monitors/                 # Background monitor configurations\n│   └── monitors.json\n├── hooks/                    # Hook configurations\n│   ├── hooks.json           # Main hook config\n│   └── security-hooks.json  # Additional hooks\n├── bin/                      # Plugin executables added to PATH\n│   └── my-tool               # Invokable as bare command in Bash tool\n├── settings.json            # Default settings for the plugin\n├── .mcp.json                # MCP server definitions\n├── .lsp.json                # LSP server configurations\n├── scripts/                 # Hook and utility scripts\n│   ├── security-scan.sh\n│   ├── format-code.py\n│   └── deploy.js\n├── LICENSE                  # License file\n└── CHANGELOG.md             # Version history\n```\n\n<Warning>\n  The `.claude-plugin/` directory contains the `plugin.json` file. All other directories (commands/, agents/, skills/, output-styles/, themes/, monitors/, hooks/) must be at the plugin root, not inside `.claude-plugin/`.\n</Warning>\n\nA `CLAUDE.md` file at the plugin root is not loaded as project context. Plugins contribute context through skills, agents, and hooks rather than CLAUDE.md. To ship instructions that load into Claude's context, put them in a [skill](#skills).\n\n### File locations reference\n\n| Component         | Default Location             | Purpose                                                                                                                                                                                    |\n| :---------------- | :--------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |\n| **Manifest**      | `.claude-plugin/plugin.json` | Plugin metadata and configuration (optional)                                                                                                                                               |\n| **Skills**        | `skills/`                    | Skills with `<name>/SKILL.md` structure                                                                                                                                                    |\n| **Commands**      | `commands/`                  | Skills as flat Markdown files. On the next Claude Code session it loads automatically as `<name>@skills-dir` and appears in `/plugin` and `claude plugin list` with no install step.\n\nSee [Skills-directory plugins](#skills-directory-plugins) for scope and trust requirements.\n\n```bash theme={null}\nclaude plugin init <name> [options]\n```\n\n**Arguments:**\n\n* `<name>`: Plugin name. Becomes the skill namespace and the directory name under `~/.claude/skills/`, so it cannot contain spaces or path separators.\n\n**Options:**\n\n| Option                   | Description                                                                                                         | Default                 |\n| :----------------------- | :------------------------------------------------------------------------------------------------------------------ | :---------------------- |\n| `--description <text>`   | Manifest description                                                                                                |                         |\n| `--author <name>`        | Author name                                                                                                         | `git config user.name`  |\n| `--author-email <email>` | Author email                                                                                                        | `git config user.email` |\n| `--with <components...>` | Also scaffold component folders. Valid values: `skills`, `agents`, `hooks`, `mcp`, `lsp`, `output-style`, `channel` |                         |\n| `-f, --force`            | Overwrite an existing `.claude-plugin/` at the target                                                               |                         |\n| `-h, --help`             | Display help for command                                                                                            |                         |\n\n**Aliases:** `new`\n\nEach `--with` value adds a starter file for that component, ready to edit:\n\n| Component      | What it scaffolds                                                                                         |\n| :------------- | :-------------------------------------------------------------------------------------------------------- |\n| `skills`       | An extra namespaced `<name>:example` skill alongside the default one                                      |\n| `agents`       | An `agents/` subagent definition                                                                          |\n| `hooks`        | A `hooks/hooks.json` with a sample event handler                                                          |\n| `mcp`          | A `.mcp.json` with HTTP and stdio server examples                                                         |\n| `lsp`          | A `.lsp.json` language-server example                                                                     |\n| `output-style` | An `output-styles/<name>.md` that applies automatically while the plugin is enabled                       |\n| `channel`      | An MCP-based [channel](/en/channels): a stdio server (`server.ts`), its `.mcp.json`, and a `package.json` |\n\nThe scaffolded plugin uses the `@skills-dir` source rather than a marketplace. Admins can block this source with `strictKnownMarketplaces` or by adding `{\"source\": \"skills-dir\"}` to `blockedMarketplaces` in [managed settings](/en/plugin-marketplaces#managed-marketplace-restrictions). When blocked, `plugin init` fails before writing.\n\n**Examples:**\n\n```bash theme={null}\n# Scaffold a minimal plugin\nclaude plugin init my-helper\n\n# Scaffold with skill and hook folders\nclaude plugin init my-helper --with skills hooks\n\n# Overwrite an existing scaffold\nclaude plugin init my-helper --force\n```\n\n### plugin install\n\nInstall a plugin from available marketplaces.\n\n```bash theme={null}\nclaude plugin install <plugin> [options]\n```\n\n**Arguments:**\n\n* `<plugin>`: Plugin name or `plugin-name@marketplace-name` for a specific marketplace\n\n**Options:**\n\n| Option                | Description                                       | Default |\n| :-------------------- | :------------------------------------------------ | :------ |\n| `-s, --scope <scope>` | Installation scope: `user`, `project`, or `local` | `user`  |\n| `-h, --help`          | Display help for command                          |         |\n\nScope determines which settings file the installed plugin is added to. For example, `--scope project` writes to `enabledPlugins` in .claude/settings.json, making the plugin available to everyone who clones the project repository.\n\n**Examples:**\n\n```bash theme={null}\n# Install to user scope (default)\nclaude plugin install formatter@my-marketplace\n\n# Install to project scope (shared with team)\nclaude plugin install formatter@my-marketplace --scope project\n\n# Install to local scope (gitignored)\nclaude plugin install formatter@my-marketplace --scope local\n```\n\n### plugin uninstall\n\nRemove an installed plugin.\n\n```bash theme={null}\nclaude plugin uninstall <plugin> [options]\n```\n\n**Arguments:**\n\n* `<plugin>`: Plugin name or `plugin-name@marketplace-name`\n\n**Options:**\n\n| Option                | Description                                                                                              | Default |\n| :-------------------- | :------------------------------------------------------------------------------------------------------- | :------ |\n| `-s, --scope <scope>` | Uninstall from scope: `user`, `project`, or `local`                                                      | `user`  |\n| `--keep-data`         | Preserve the plugin's [persistent data directory](#persistent-data-directory)                            |         |\n| `--prune`             | Also remove auto-installed dependencies that no other plugin requires. Dependencies that Claude Code pulled in to satisfy another plugin's [`dependencies`](/en/plugin-dependencies) field are removed; plugins you installed directly are never touched.\n\n```bash theme={null}\nclaude plugin prune [options]\n```\n\n**Options:**\n\n| Option                | Description                                                              | Default |\n| :-------------------- | :----------------------------------------------------------------------- | :------ |\n| `-s, --scope <scope>` | Prune at scope: `user`, `project`, or `local`                            | `user`  |\n| `--dry-run`           | List what would be removed without removing anything                     |         |\n| `-y, --yes`           | Skip the confirmation prompt. Requires `--json` |         |\n| `-h, --help`  | Display help for command                                       |         |\n\nWithin an interactive session, `/plugin list` prints the same listing inline. The interactive form accepts `--enabled` or `--disabled` to show only plugins in that state, and `ls` as a shorthand for `list`.\n\n### plugin details\n\nShow a plugin's component inventory and projected token cost. The output lists all components the plugin contributes, grouped as Skills, Agents, Hooks, MCP servers, and LSP servers, along with an estimate of how many tokens it adds to each session. The Skills group includes both `skills/` and `commands/` entries.\n\n```bash theme={null}\nclaude plugin details <name>\n```\n\n**Arguments:**\n\n* `<name>`: Plugin name or `plugin-name@marketplace-name`\n\n**Options:**\n\n| Option       | Description              | Default |\n| :----------- | :----------------------- | :------ |\n| `-h, --help` | Display help for command |         |\n\nThe output shows two cost figures for each component:\n\n* **Always-on:** tokens added to every session by the plugin's listing text, such as skill descriptions, agent descriptions, and command names, regardless of whether any component fires.\n* **On-invoke:** tokens a component costs when it fires. Shown per component, not as a plugin total, because a typical session invokes only a subset of components.\n\nThis example shows what the output looks like for a plugin with two skills:\n\n```\ndependency-guard 1.2.0\n  Dependency analysis for Claude Code sessions\n  Source: dependency-guard@example-marketplace\n\nComponent inventory\n  Skills (2)  scan-dependencies, review-changes\n  Agents (0)\n  Hooks (1)  (harness-only — no model context cost)\n  MCP servers (0)\n  LSP servers (0)\n\nProjected token cost\n  Always-on:   ~180 tok   added to every session\n\nPer-component (rounded)\n  component            always-on  on-invoke\n  scan-dependencies        ~100      ~2400\n  review-changes            ~80      ~1800\n\n  On-invoke cost is paid each time a skill or agent fires.\n  Token counts are estimates and may differ from actual usage.\n```\n\nThe always-on total is computed via the `count_tokens` API for your active model. Per-component numbers are proportionally scaled from that total. If the API is unreachable, the command falls back to a character-based estimate.\n\n### plugin tag\n\nCreate a release git tag for the plugin in the current directory. Description                                                                | Default |\n| :------------ | :------------------------------------------------------------------------- | :------ |\n|
